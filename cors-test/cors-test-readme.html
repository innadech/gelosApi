🧪 CORS Testing Lab

Prove that CORS works by running the API and client on different ports

What This Lab Proves

In Postman, CORS doesn't exist — Postman is not a browser. To see CORS in action, you need a browser making requests from a different origin. This lab gives you exactly that:

CORS Test Client http://localhost:5500
A web page with buttons that call your API
≠
Your Gelos API http://localhost:3000
Express + MongoDB + CORS (Session 3)
DIFFERENT PORTS = DIFFERENT ORIGINS. http://localhost:5500 and http://localhost:3000 are different origins because the port numbers differ. The browser will enforce CORS on every request from 5500 → 3000, including pre-flight OPTIONS requests that you can watch in DevTools.

Setup — Step by Step

Step 1: Start your Gelos API (port 3000)

Open Terminal 1 (Command Prompt, PowerShell, or VS Code terminal).
Navigate to your gelos-api project folder.
Start the server:
Terminal 1Start your API
cd C:\gelos-api
npm run dev

You should see:

Output
Server running on http://localhost:3000
API docs:  http://localhost:3000/api-docs
Connected to MongoDB
KEEP THIS TERMINAL OPEN. The API must stay running the entire time you're testing.

Step 2: Start the CORS test client (port 5500)

Open Terminal 2 (a second terminal window — not the same one).
Navigate to where you saved the CORS test files.
Run the test server:
Terminal 2Start the CORS test client
cd C:\cors-test
node cors-test-server.js

You should see:

Output
  🧪 CORS Test Client running at:
     http://localhost:5500

  Your API should be running at:
     http://localhost:3000

  Different ports = different origins = CORS enforced!
KEEP THIS TERMINAL OPEN TOO. You now have two servers running side by side.

Step 3: Open the test client in your browser

Open Chrome or Firefox.
Go to: http://localhost:5500
Open DevTools: press F12 (or right-click → Inspect).
Click the Network tab in DevTools.
Make sure the Network tab is recording (red dot visible).
YOU SHOULD NOW SEE: The CORS Test Client page with origin boxes showing port 5500 and port 3000, login fields, and method buttons. DevTools Network tab is open and ready to capture requests.

Testing — Run Each Scenario

Test A: GET all orders (simple request)

Click GET /api/orders.
Watch the Network tab — you should see the GET request appear.
Check the log panel — CORS headers should appear in yellow: access-control-allow-origin: *
GET is a "simple request" — the browser may skip pre-flight and just check the response headers.
WHAT YOU'RE SEEING: The browser sent a GET from port 5500 to port 3000. Because these are different origins, the browser checks the response for Access-Control-Allow-Origin. Your cors(corsOptions) adds this header automatically. If it weren't there, the browser would block the response.
SCREENSHOT: DevTools Network tab showing GET request with CORS headers

Test B: GET all orders

Click GET /api/orders.
Watch the Network tab — you may see:
  • OPTIONS /api/orders → 204 (pre-flight — because Authorization header is custom)
  • GET /api/orders → 200 OK
Check the log — CORS headers should appear in yellow: access-control-allow-origin: *

Test C: POST new order (create + pre-flight)

Click POST /api/orders.
Network tab shows: OPTIONS → 204, then POST → 201 Created.
Log shows the created order (OrderNo 9998) + CORS headers.

Test D: PUT (replace order)

Click PUT /api/orders/9998.
Network tab: OPTIONS → 204, then PUT → 200 OK.
PUT always triggers pre-flight — it's not a "simple method".

Test E: PATCH (partial update)

Click PATCH /api/orders/1001.
Network tab: OPTIONS → 204, then PATCH → 200 OK.
Only the Product Quantity field is updated (the rest stays the same).

Test F: DELETE

Click DELETE /api/orders/9998.
Network tab: OPTIONS → 204, then DELETE → 200 OK.
The test order #9998 is removed (cleaning up after Test C/D).

Test G: HEAD (metadata only)

Click HEAD /api/orders.
Response body is empty — that's correct for HEAD.
CORS headers should still appear on the response.

🚫 Test H: Prove CORS is Required (Break It!)

This is the most important test — it proves that without cors(), everything breaks.

Open your gelos-api/server.js in VS Code.
Find this line: app.use(cors(corsOptions));
Comment it out by adding // at the start:
server.jsComment out CORS
// app.use(cors(corsOptions));   ← add // to disable CORS
Save the file. Nodemon restarts your server automatically.
Go back to the CORS test client (http://localhost:5500).
Click GET /api/orders.
WATCH IT FAIL. The log will show a red CORS error.
Open the Console tab in DevTools — you'll see the full error:
Browser ConsoleThe CORS error
Access to fetch at 'http://localhost:3000/api/orders'
from origin 'http://localhost:5500'
has been blocked by CORS policy:
No 'Access-Control-Allow-Origin' header is present
on the requested resource.
THIS IS THE PROOF. Without app.use(cors(corsOptions)), the server doesn't send CORS headers. The browser blocks the response. The data never reaches the client. This is exactly what would happen if a malicious website tried to call your API — the browser protects the user.
SCREENSHOT: Browser console showing CORS error after disabling cors()

Fix it

Go back to server.js.
Remove the // to re-enable CORS:
server.jsRe-enable CORS
app.use(cors(corsOptions));   ← remove // to enable CORS
Save. Nodemon restarts.
Click GET /api/orders again in the test client.
It works again. 200 OK with CORS headers.
ONE LINE. That single app.use(cors(corsOptions)) is the difference between a working API and a completely blocked one. This is what Session 3 configured.

What to Look For in DevTools

Network tab — pre-flight requests

What you seeWhat it means
OPTIONS request before POSTBrowser sent pre-flight — your API has JSON body + Auth header
OPTIONS status: 204Pre-flight accepted — server says "yes, that method and those headers are allowed"
OPTIONS response headers include Access-Control-Allow-MethodsServer lists which methods are permitted
The actual POST follows the OPTIONSBrowser approved the pre-flight → sent the real request
Only GET appears (no OPTIONS before it)GET is a "simple request" — browser skipped pre-flight (unless Auth header triggers it)

Network tab — click on the OPTIONS request

Click the OPTIONS request in the Network tab. Look at the Response Headers:

HeaderExpected valueMeaning
Access-Control-Allow-Origin*Any origin can call this API
Access-Control-Allow-MethodsGET,HEAD,PUT,PATCH,POST,DELETEAll these methods are permitted
Access-Control-Allow-HeadersContent-Type,AuthorizationJSON bodies (and later JWT tokens) are allowed

Console tab — CORS errors

If CORS is broken, the Console tab shows the exact error. Common messages:

Error messageCauseFix
"No 'Access-Control-Allow-Origin' header"cors() not enabled in server.jsAdd app.use(cors(corsOptions))
"Method PUT not allowed"PUT not in your methods arrayAdd PUT to corsOptions.methods
"Header 'Authorization' not allowed"Authorization not in allowedHeadersAdd Authorization to corsOptions.allowedHeaders
"Failed to fetch" (no details)Server not running or wrong portCheck npm run dev is running on port 3000
